Laboratorio - Hydra

pentesting laboratorio docker ssh wordlist

El objetivo de este laboratorio es comprender el funcionamiento de Hydra dentro de un entorno completamente controlado, observando cómo se realizan las pruebas de autenticación y cómo responden los servicios frente a intentos repetidos de inicio de sesión.


1. Preparación del entorno

Antes de comenzar es recomendable preparar un entorno aislado para evitar interferir con otros equipos de la red.

  • Máquina virtual (VirtualBox, VMware, etc.).
  • Red interna o laboratorio aislado.
  • Kali Linux o distribución similar.
  • Servicio SSH instalado y funcionando.

2. Máquinas vulnerables

Para realizar las pruebas es recomendable utilizar máquinas preparadas para laboratorio o contenedores Docker con servicios vulnerables.

  • DVWA
  • Metasploitable
  • OWASP Broken Web Apps
  • Contenedores Docker de prácticas

3. Crear usuarios de prueba

Configura varias cuentas con contraseñas sencillas para comprobar cómo Hydra identifica credenciales débiles.

Ejemplos:

  • admin / admin123
  • usuario / 12345678
  • paquito / paquito06
  • manolo / manolopiedra

4. Crear las listas de prueba

Hydra utiliza listas de usuarios y contraseñas para generar automáticamente todas las combinaciones posibles.

usuarios.txt

  • admin
  • usuario1
  • pepe
  • maria
  • test

wordlist.txt

  • 123456
  • password
  • qwerty
  • admin123
  • letmein

5. Ejecutar la auditoría

Una vez preparado el laboratorio puede ejecutarse Hydra contra el servicio SSH del equipo de pruebas. 

└─$ hydra -L usuarios.txt -P wordlist.txt ssh://192.168.1.100

6. Observar los resultados

Durante la ejecución conviene prestar atención a varios aspectos:

  • Número de intentos realizados.
  • Velocidad de ejecución.
  • Mensajes del servidor SSH.
  • Credenciales válidas encontradas.
  • Posibles bloqueos de cuentas.
  • Registros generados en el sistema.

7. Medidas defensivas

Tras finalizar la práctica resulta recomendable repetir el laboratorio aplicando medidas de protección para comprobar cómo disminuye la eficacia de la auditoría.

  • Contraseñas robustas.
  • Autenticación multifactor (MFA).
  • Bloqueo temporal tras varios intentos.
  • Fail2Ban.
  • Monitorización de logs.

Conclusión / Resumen del mini proyecto

Este laboratorio permite comprender que la principal debilidad no suele encontrarse en la herramienta utilizada, sino en la calidad de las credenciales y en las políticas de autenticación implementadas por el sistema.
